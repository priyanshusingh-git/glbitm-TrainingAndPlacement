The landing page currently has a fundamental misunderstanding of mobile-first design. Making every grid go to 1-column on mobile is not mobile-first — it is desktop shrunk down. Industry-standard mobile design uses different visual patterns for mobile, not just fewer columns.

1.1  What "Mobile-First" Actually Means for Landing Pages
Mobile-first CSS (min-width queries) is correct — but mobile-first design goes further. It means the mobile layout is a distinct, purpose-built experience — not the desktop layout with fewer columns. On desktop, 6 cards in 2 rows is scannable. On mobile, 6 cards in a single column forces the user to scroll through 6 full cards to understand a section that could be communicated in 2 rows of 3 compact items.

Misconception
Reality
1-col mobile = mobile-first
1-col is often the wrong treatment. Mobile-first means choosing the right pattern for a small screen, which is often 2-col compact or horizontal scroll.
Show all content on mobile
Progressive disclosure: show the summary on mobile, let users tap to see more. Every major app does this.
Responsive means same content, different layout
Responsive means different content density per breakpoint. Mobile gets scannable, desktop gets detailed.
Section padding should match desktop
Mobile sections need tighter vertical padding to reduce scroll distance without losing breathing room.

3.  The 5 Industry-Standard Mobile Patterns
These are the five layout patterns used across every major app and website for handling content-dense sections on mobile. Each pattern is applied to specific sections of the GLBITM portal based on the content type.

Pattern 1 — 2-Col Compact Grid (Most Used)
Default for all feature grids on mobile. Show icon + title only. Hide body text. User taps to expand if interested. This is how Razorpay, Stripe, Linear, Notion handle feature sections on mobile.

When to use
When NOT to use
Cards with a clear icon + short title (≤40 chars)
Cards where the body text is essential to understand what is being offered
6+ items that would otherwise stack very tall
Cards with complex data that cannot be compressed to a title
Items that are scannable at a glance
When each card has very different visual weight

/* Pattern 1 — 2-col compact, body hidden on mobile */
.pg { display:grid; grid-template-columns:repeat(2,1fr); gap:10px; }
.pc p { display:none; }           /* hide body on mobile */
.pc.expanded p { display:block; } /* show on tap */
 
/* Restore full layout at 1024px */
@media(min-width:1024px) {
  .pg { grid-template-columns:repeat(3,1fr); gap:18px; }
  .pc p { display:block; }        /* always show on desktop */
}

Applied to: Process cards (.pg), Training modules (.tm), CDC Skills (.skg), Guidance cards (.gg), Team cards (.teamg), Advisory (.ag)

Pattern 2 — 2×2 Stat Grid (Always)
Stat/number blocks are never 1-col on mobile. Numbers are short (2–5 characters). A 2×2 grid fits perfectly on 360px. This is how Paytm, PhonePe, Groww, LinkedIn display metrics — always 2 across, never single column.

Why 2×2 is always correct for stats

A stat block contains: rule (22px) + number (38–52px) + label (10px) + sublabel (10px) = ~90–110px tall — fits 2-col at 360px comfortably
Column width at 2-col: (360 - 32px padding - 1px border) / 2 = ~163px per column. A 5-char number at 38px fits easily.

/* Pattern 2 — 2×2 stats — remove 1-col base, start at 2-col */
.sg { display:grid; grid-template-columns:repeat(2,1fr); }
.sc { padding:20px 14px; }
.sc:nth-child(odd) { border-right:1px solid rgba(255,255,255,.06); }
.sc:nth-child(3),.sc:nth-child(4) { border-top:1px solid rgba(255,255,255,.06); }
 
/* Scale up at 1024px */
@media(min-width:1024px) {
  .sg { grid-template-columns:repeat(4,1fr); }
  .sc { padding:40px 32px; }
}

Pattern 3 — Horizontal Scroll Strip
For category lists, filters, logo strips, and any content where items are equal in importance and the user scans horizontally rather than vertically. Key: add a visible scroll indicator (fade + partial next item) so users know it is scrollable. Without this, many users never discover the scroll.

/* Pattern 3 — horizontal scroll with peek indicator */
.rc-strip {
  display:flex;
  overflow-x:auto;
  -webkit-overflow-scrolling:touch;
  scrollbar-width:none;
  gap:10px;
  padding-bottom:4px;
  /* Fade right edge to hint scrollability */
  mask-image:linear-gradient(to right, black 85%, transparent);
  -webkit-mask-image:linear-gradient(to right, black 85%, transparent);
}
.rc-strip .rcat { min-width:200px; flex-shrink:0; }
 
/* Remove mask and restore grid at 640px */
@media(min-width:640px) {
  .rc-strip { display:grid; grid-template-columns:repeat(2,1fr); mask-image:none; }
  .rc-strip .rcat { min-width:auto; }
}

Applied to: Recruiter category strip (.rc). Logo grid stays as grid (3-col at base).

Pattern 4 — Progressive Disclosure (Collapse/Expand)
For text-heavy content that users need to reference but do not need to read all at once. Show the heading and first item. Hide the rest behind a "Show more" button. This is used universally for terms, policy, FAQs, and feature details on mobile.

/* Pattern 4 — collapse/expand for policy cards */
.pl-body { max-height:0; overflow:hidden; transition:max-height .3s ease; }
.pl.expanded .pl-body { max-height:500px; }
.pl-toggle {
  display:flex; align-items:center; justify-content:space-between;
  cursor:pointer; padding:16px 0; min-height:44px;
}
.pl-toggle::after { content:"▸"; color:var(--am-d); transition:transform .2s; }
.pl.expanded .pl-toggle::after { transform:rotate(90deg); }
 
/* Always expanded on desktop */
@media(min-width:1024px) {
  .pl-body { max-height:none !important; }
  .pl-toggle::after { display:none; }
}

Applied to: Placement Policy cards (.polg). Each card title is a tap target. Body collapses by default. First card is pre-expanded.

Pattern 5 — 3-Col Logo/Icon Grid (Minimum)
Small logo tiles and icon badges are never 2-col on mobile — they are compact enough for 3-col even at 360px. Each tile is: icon (36×36px) + name (10.5px) + sector (9px) = ~80px tall. 3-col at 360px gives 107px per column — plenty of space.

/* Pattern 5 — 3-col logo grid at base (not 2-col) */
.lg { display:grid; grid-template-columns:repeat(3,1fr); gap:8px; }
 
@media(min-width:480px) { .lg { grid-template-columns:repeat(4,1fr); } }
@media(min-width:1024px){ .lg { grid-template-columns:repeat(6,1fr); } }

Applied to: Recruiter logo grid (.lg). Replaces current 2-col base.

// JS — accordion expand for process cards
document.querySelectorAll('.pc').forEach(function(card) {
  card.addEventListener('click', function() {
    if(window.innerWidth >= 1024) return; // desktop always open
    var wasOpen = card.classList.contains('expanded');
    document.querySelectorAll('.pc').forEach(c => c.classList.remove('expanded'));
    if(!wasOpen) card.classList.add('expanded');
    card.setAttribute('aria-expanded', !wasOpen);
  });
  card.setAttribute('role','button');
  card.setAttribute('aria-expanded','false');
  card.setAttribute('tabindex','0');
});

6.  JavaScript Interactions Required
Three JS interactions are needed for the mobile-specific expand/collapse patterns. All interactions must be disabled at 1024px and above (desktop always shows full content).

6.1  Accordion — Process Cards
// Process card accordion (single expand at a time)
(function() {
  var cards = document.querySelectorAll('.pc');
  cards.forEach(function(card) {
    card.setAttribute('role','button');
    card.setAttribute('tabindex','0');
    card.setAttribute('aria-expanded','false');
    function toggle() {
      if(window.innerWidth >= 1024) return;
      var open = card.classList.contains('expanded');
      cards.forEach(c => { c.classList.remove('expanded'); c.setAttribute('aria-expanded','false'); });
      if(!open) { card.classList.add('expanded'); card.setAttribute('aria-expanded','true'); }
    }
    card.addEventListener('click', toggle);
    card.addEventListener('keydown', function(e) { if(e.key==='Enter'||e.key===' ') { e.preventDefault(); toggle(); } });
  });
})();

6.3  Collapse/Expand — Policy Cards
// Policy collapse/expand (independent — any can be open)
(function() {
  var cards = document.querySelectorAll('.pl');
  cards.forEach(function(card, i) {
    var body = card.querySelector('.pl-body');
    var h3 = card.querySelector('h3');
    if(!body || !h3) return;
    h3.setAttribute('tabindex','0');
    h3.setAttribute('role','button');
    // First card pre-expanded
    if(i === 0) { card.classList.add('open'); }
    else { body.classList.add('collapsed'); }
    function toggle() {
      if(window.innerWidth >= 1024) return;
      card.classList.toggle('open');
      body.classList.toggle('collapsed');
    }
    h3.addEventListener('click', toggle);
    h3.addEventListener('keydown', function(e) { if(e.key==='Enter'||e.key===' ') { e.preventDefault(); toggle(); } });
  });
})();
